[A3S 프로젝트 백엔드 피드백 리포트 - 통합본]

  1. 주문 목록 조회 N+1 문제
   * 상태: [해결 완료]
   * 확인: 주문 목록 조회 시 각 주문마다 결제 정보를 별도로 조회하여 1+N 쿼리가 발생하던 지점입니다.
   * 분석: 주문 데이터가 늘어날수록 성능이 기하급수적으로 저하되는 구조였습니다.
   * 개선: Batch Fetching(IN 절 조회) 및 메모리 Map 매칭 방식을 도입하여 주문 수와 관계없이 쿼리를 2번으로 고정(O(1))했습니다.

  2. 결제 웹훅 (Webhook) 멱등성 보장
   * 상태: [해결 완료]
   * 확인: WebhookService.java에서 webhookRepository.existsByWebhookUuidAndStatus를 통해 이미 처리된 웹훅인지 먼저 검증하고 있습니다. 
   * 분석: PG사는 네트워크 상태에 따라 동일한 결제 알림을 중복으로 보낼 수 있어 중복 처리를 막는 것이 필수적입니다.
   * 확인: Webhook 엔티티를 별도로 생성하여 상태(PROCESSED, FAILED)를 관리하는 방식은 아주 훌륭한 멱등성 보장 전략입니다.

  3. 보상 트랜잭션의 구현 (Partial Failure 대응)
   * 상태: [해결 완료]
   * 확인: PaymentFailureHandler 및 PaymentRollback(REQUIRES_NEW)을 통해 예외 발생 시 실패 이력을 남깁니다.
   * 분석: 외부 API 호출과 DB 트랜잭션이 섞인 환경에서 시스템 오류 시 데이터 정합성을 유지해야 합니다.
   * 확인: 비즈니스 예외 발생 시 DB 상태를 안전하게 롤백하고 실패 기록을 남기는 보상 트랜잭션 구조가 이미 구현되어 있음을 확인했습니다.

  4. IDOR 검증의 전수 적용 및 일관성 유지
   * 상태: [해결 완료]
   * 확인: 주문 상세 조회 및 결제 확정 시점에 userId 소유권 검증이 이루어지고 있습니다.
   * 분석: PathVariable로 받는 ID가 현재 로그인 사용자의 것인지 검증하지 않으면 보안 취약점이 발생합니다.
   * 확인: JWT의 userId 정보를 바탕으로 소유권 검증 로직(findByIdAndUser_Id 등)이 견고하게 적용되어 있음을 확인했습니다.

  5. Refresh Token Rotation (RTR) 도입
   * 상태: [해결 완료]
   * 확인: AuthService.reissue에서 토큰 재발급 시 기존 Refresh Token을 무효화하고 새 토큰을 발급합니다.
   * 분석: 리프레시 토큰 탈취 시 무한 재발급을 방지하기 위한 핵심 보안 조치입니다.
   * 확인: RTR 로직이 안정적으로 구현되어 서비스 보안성이 확보되었음을 확인했습니다.

  6. 비관적 락(Pessimistic Lock) 사용 관련
   * 상태: [해결 완료]
   * 확인: PointService 등에서 유저 포인트 변동 시 비관적 락(SELECT FOR UPDATE)을 사용 중입니다.
   * 분석: 동시성 처리는 확실하나 트래픽 증가 시 성능 저하 및 데드락 가능성이 있습니다.
   * 확인: 팀 정책에 따른 의도적인 선택임을 확인하였으며, 현재 구조에서 동시성 보장을 위해 적절히 사용됨을 검증했습니다.

  7. 대량 데이터 조회 시 페이징 (Pagination) 미적용
   * 상태: [전략적 유지]
   * 확인: 상품 목록 및 포인트 내역 조회 시 전체 리스트를 반환하고 있습니다.
   * 분석: 데이터 축적 시 메모리 부하(OOM) 및 응답 속도 저하 위험이 있습니다.
   * 확인: 프론트엔드 요구사항 및 과제 환경에 맞춘 전략적 선택임을 확인하였으며, 현재 규모에서는 성능상 문제가 없음을 검증했습니다.

  8. 포인트 소멸 스케줄러 부하
   * 상태: [개선 필요]
   * 확인: PointExpireScheduler.java의 expirePoints() 메서드에서 만료된 모든 거래를 List<PointTransaction>으로 한꺼번에 가져와 루프를 돌립니다.
   * 분석: 현재는 데이터가 적어 괜찮지만, 만료 대상이 수만 건일 경우 이 스케줄러가 도는 동안 DB 커넥션과 CPU를 독점하게 됩니다. 루프 내부에서 userRepository.findWithLockById를 호출하고 있어, 스케줄러가 도는 동안 사용자들이 로그인을 못 하거나 주문을 못 하는 상황이 발생할 수 있습니다.
   * 제안: Pageable을 사용하여 500건씩 끊어서 처리(Batching)하거나, 처리 간격을 두는 방식이 필요합니다.

  9. 환불 시 포인트 회수 시나리오
   * 상태: [잠재적 리스크]
   * 확인: RefundService.java에서 적립된 포인트를 회수(pointService.cancelEarnedPoint)합니다.
   * 분석: PointService.cancelEarnedPoint 내부에서는 user.usePoint(amount)를 호출하는데, 만약 사용자가 적립된 포인트를 이미 다 써서 잔액이 부족하면 PointException이 발생하며 환불 전체가 롤백됩니다.
   * 리스크: "포인트를 써버려서 환불이 안 되는" 상황은 비즈니스적으로 어색할 수 있습니다. (보통은 포인트 잔액을 마이너스로 만들거나, 환불할 현금에서 차감하는 정책을 취합니다.)

 10. 재고 관리 동시성 이슈 (Race Condition)
   * 상태: [개선 필요]
   * 확인: 현재 재고 확인(validateProductForOrder)은 단순 SELECT 결과에 의존하고 있습니다.
   * 분석: 수천 명이 동시에 주문할 경우 초과 판매(Overselling)가 발생할 위험이 있습니다.
   * 제안: 상품 재고 차감 시에도 비관적 락을 걸거나 원자적 업데이트(Atomic Update) 쿼리를 사용하는 전략 검토가 필요합니다.

 11. 회원 및 멤버십 조회 효율화
   * 상태: [개선 필요]
   * 확인: MembershipService 등에서 User 조회 후 다시 Membership을 조회하는 2-step 방식을 사용 중입니다.
   * 분석: 단순 조회임에도 쿼리가 2번 발생하여 네트워크 오버헤드가 발생합니다.
   * 제안: Join Fetch를 사용하여 단일 쿼리로 최적화하면 코드 간결화와 성능 향상을 기대할 수 있습니다.

   * 제안: 외부 API 호출과 DB 저장을 분리하거나, 비동기 처리 및 트랜잭션 범위 최소화 전략이 필요합니다.

 13. 엔티티 공통 필드 및 Soft Delete
   * 상태: [부분 해결]
   * 확인: BaseEntity.java에 createdAt과 modifiedAt이 잘 구현되어 있습니다.
   * 분석: 다만, 데이터의 이력 보존을 위한 Soft Delete(논리 삭제) 필드는 보이지 않습니다.
